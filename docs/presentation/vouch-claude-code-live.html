the problem

your agent forgets everything — or remembers anything.

amnesia

every session re-greps the repo, re-asks the same questions, re-learns the same conventions. you pay for the onboarding again and again.

unsupervised memory

auto-memory tools fix amnesia by letting the agent write down whatever it concluded — right or wrong. a hallucination stored today is a confident "fact" next month.

no provenance

six weeks later nobody can answer "why do we believe this?" — no source, no reviewer, no record of what it replaced.

agents should not start every session with amnesia — but they shouldn't get to write whatever they want either.README.md, vouchdev/vouch

the idea

review agent-claimed facts like you review agent-written code.

agents propose; humans approve. a claim without cited evidence is a validation error, not a warning. nothing becomes durable knowledge without passing the gate.

what it is, concretely

a folder in your repo. plaintext all the way down.

.vouch/ — after vouch init
.vouch/
├── config.yaml            # kb settings
├── audit.log.jsonl        # append-only audit (committed)
├── claims/<id>.yaml       # reviewed claims
├── pages/<id>.md          # markdown pages
├── sources/<sha256>/      # content-addressed evidence
├── proposed/<id>.yaml     # pending (gitignored)
├── decided/<id>.yaml      # approvals + rejections (committed)
└── state.db               # derived fts5 index (not committed)

git is the infrastructure

the kb diffs in prs, travels with clones, and needs no service. rejected drafts are gitignored, so they never pollute history.

the audit log is the history

every mutation is an append-only event with an actor. decided/ is the queryable summary of every approve and reject.

search is derived, never authoritative

state.db is an fts5 (optionally embedding) index rebuilt from the files. the yaml is the source of truth.

agent actions and human decisions, interleaved and attributed, in an append-only log that lives in git. when a rule changes later, supersede keeps the old claim queryable — history is never rewritten.

why devs keep it

trust the recall, because you gated the writes.

recall you can trust

kb.context serves only reviewed, cited claims — so injecting it into a fresh session raises quality instead of laundering old hallucinations.

bad writes die young

the state.db misreading lived four minutes and never entered the kb. its rejection reason is permanent documentation.

multi-agent attribution

claude code, cursor, a ci bot — each proposal and event carries its actor. "which agent claimed what" has an exact answer.

zero new infrastructure

no service, no db to host. yaml + markdown + git you already have. prs are the review ui for knowledge, too.

honest counter-cases, straight from the readme: skip vouch for solo throwaway projects, or if your team won't actually review proposals — without the gate it's a worse note app.

get started — before your agent's next session

ten minutes to a gated memory.

install + init
pipx install vouch-kb
cd your-project && vouch init
# creates .vouch/ with a starter config
# and one cited example claim

vouch install-mcp claude-code
# or paste the .mcp.json block from slide 5
your daily loop
vouch pending          # what did the agents learn?
vouch show <id>        # read one proposal
vouch approve <id>     # → durable, cited claim
vouch reject <id> --reason "…"
git add .vouch/ && git commit
